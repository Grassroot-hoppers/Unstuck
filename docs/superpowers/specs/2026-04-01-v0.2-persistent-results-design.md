# Design: v0.2 — Persistent results (Supabase) + “save this link” UX

**Status:** Draft for implementation  
**Date:** 2026-04-01  
**Scope:** ROADMAP v0.2 — permanent results, short URLs, hash fallback, no accounts.

## Goals

- Save assessment results to Supabase automatically on completion.
- Redirect to a short, stable URL: `results.html?id=<shortId>`.
- Keep hash-based URLs working when save fails or for legacy links.
- Help distracted users **not lose** their link: explicit prompt + **email-yourself** affordance.
- Add **localStorage** “last result” pointer so the same browser can reopen recent results from the home page (convenience, not source of truth).
- Stay **static**: no build step; optional small shared `.js` file to avoid duplicating Supabase logic.

## Non-goals (v0.2)

- User accounts / magic link.
- Server-sent email (Edge Function + Resend) — later.
- Changing IPIP items or scoring.

## Data flow

1. User completes `assessment.html` → scores computed (unchanged).
2. Client inserts one row into `results` (schema per `docs/supabase-setup.md`).
3. On success: redirect to `results.html?id=<id>&new=1`.
4. On failure: redirect to `results.html#<base64>` (existing behavior).

`new=1` is **not stored in the database**; it only signals the UI to show the “save your link” modal once for this arrival from completion.

## “Save this link” modal (pop-up)

**Do not use `window.alert` / `confirm`.** Use a small **in-page modal** (focus trap optional v0.2 minimum: focus primary button and Escape to close if feasible) styled with existing tokens in `src/styles.css` (cream/forest/terracotta).

**When it shows**

- Only when **both** are true:
  - URL has `?id=<valid load>` (Supabase-backed result), and
  - `new=1` is present **or** (fallback) first paint after redirect from assessment if we ever drop `new=1` — **primary rule: assessment must append `new=1` on successful save.**

**After the user dismisses**

- Strip `new=1` from the address bar with `history.replaceState` so refreshes don’t re-show the modal.
- Do **not** use a blocking second modal; one dialog is enough.

**Content (copy can be refined during implementation)**

- Headline: e.g. “Keep this link safe” / “Don’t lose your results.”
- Short body: your results live at this link; save it somewhere you’ll find again.
- **Primary:** “Copy link” (uses Clipboard API with fallback: select a read-only input).
- **Secondary:** “Email me this link” — `mailto:` with `subject` + `body` containing the full permalink (`window.location.origin` + path + `?id=` only, no `new=1`). Encode with `encodeURIComponent`.
- **Tertiary:** “Got it” / dismiss.

**Accessibility**

- `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at headline.
- Visible focus states consistent with site.

## Last-result convenience (localStorage)

- After a **successful** save (or after successfully loading from `?id=` if we only set it on save — **prefer: set only when we created the row in assessment** to avoid overwriting with someone else’s shared link): set `localStorage['unstuck_last_result_id'] = id` (key name exact in implementation; prefix avoids collisions).
- On `index.html` (and optionally `assessment.html` entry): if key exists, show a calm secondary CTA: “Open your last results” → `results.html?id=…`.
- If `id` load fails later, clear or ignore the key (implementation detail: try load once, on 404/error remove key).

## Consent / architecture note

- Current `ARCHITECTURE.md` invariant #4 says no server calls without **explicit** consent. For v0.2 we adopt **automatic save** with **plain-language transparency** (results page / modal copy: stored so this link keeps working). **Follow-up task:** update `ARCHITECTURE.md` invariant wording to match this version so docs don’t contradict behavior.

## Implementation boundaries

- **Shared module:** `src/supabase-results.js` (or similar) exporting: init client (reads URL + anon key from `window.__UNSTUCK_SUPABASE__` or script-tag constants — TBD single pattern), `generateShortId`, `saveResultsRow`, helpers for last-result key. Include via `<script src="...">` before inline scripts on both pages.
- **Collision handling:** short IDs may collide; on unique violation, regenerate and retry (cap retries, then fallback to hash).
- **Security model:** unchanged — ID is capability; RLS as in setup doc.

## Verification

- Complete assessment → `?id=` + modal → copy + mailto smoke test.
- New browser/incognito: same `?id=` loads; modal does not fire without `new=1`.
- Save failure → hash URL still works.
- Home shows “last results” when key set; wrong id clears gracefully.
